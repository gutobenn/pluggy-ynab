# pluggy-ynab

Syncs Brazilian bank transactions to YNAB using Pluggy as the data provider.

## How to run

```bash
python sync.py                       # bare + interactive terminal -> opens the menu; otherwise (cron/piped) imports last 14 days
python sync.py --from 2026-01-01
python sync.py --dry-run             # fetch + summarize everything, don't save to YNAB
python sync.py -v                    # also print the per-account transaction dump (hidden by default; --show-transactions/--verbose)
python sync.py --dry-run --debug     # also print per-page fetch counts vs Pluggy's total
python sync.py --reconcile           # show only the balance reconciliation (Pluggy vs YNAB balances, no import), then exit
python sync.py --doctor              # check each connection (auth + Pluggy reachability + freshness), then exit
python sync.py --list-accounts <ITEM_ID>   # discover Pluggy account ids for accounts.json, then exit
python sync.py --update-investments  # post the YNABâ†”Pluggy diff per investment account as a "Rendimento" txn
python sync.py --no-menu             # force the non-interactive sync even in a terminal; --menu forces the menu anywhere
```

Requires a `.env` (secrets, see `.env.example`) and an `accounts.json` (list of accounts to sync, see `accounts.example.json`) at the project root. `mappings.json` is optional (see `mappings.example.json`).

The UI is built on **rich** (progress, panels, tables) + **questionary** (menus, with the highlighted option styled in cyan). Launched bare in a TTY, `sync.py` shows an **interactive menu**: Sync transactions / Update investments / Balance reconciliation / Check connections / Quit. "Sync" runs an account-selection checklist + a post-sync "show transaction details?" prompt. "Update investments" is **investment-only** (fetches just the investment accounts and posts the Rendimento diff â€” the `--update-investments` *flag* instead layers the true-up onto a full sync of all accounts, for cron). Dry-run (`--dry-run`) and account discovery (`--list-accounts`) are flag-only (not in the menu). With no TTY (cron, piped) or any action flag, it runs non-interactively. The per-account transaction dump is **opt-in** (`-v` or the menu prompt) â€” `--dry-run` means "don't save" only and no longer implies `--debug`. During the concurrent fetch a live per-account line (spinner â†’ âœ“/âœ— + count) is shown; the run ends with an Import Summary panel.

Every run ends with a balance-reconciliation table (grouped by type: checking / credit cards / investments) comparing each account's current Pluggy balance to its YNAB balance (cleared + uncleared), flagging match/MISMATCH (credit cards sign-inverted). Transaction fetching paginates through all pages (Pluggy caps a page at 500). Investment accounts are **report-only by default** (Pluggy exposes only position balances, no transactions); with `--update-investments` the per-account difference is posted as a "Rendimento" adjustment via `YNABTransactionImporter.add_adjustment`. The `import_id` is `REND-<date>-<time>-<acct8>` (**unique per run** â€” YNAB permanently remembers import_ids and rejects re-imports of a used one as a duplicate even after deletion, so a fixed daily id could never recreate a deleted Rendimento). Idempotency comes from the diff check instead: once an account is trued up, the next run sees `diff â‰ˆ 0` and posts nothing (and after a save the cached YNAB balances are refreshed so a second update in the same menu session doesn't double-post).

## Project structure

- `sync.py` - Entry point / orchestration only (all presentation lives in `ui.py`). A `Session` holds creds + config and lazily connects the YNAB client (once per process, reused across menu iterations). `main()` parses args and dispatches: `--list-accounts`/`--doctor` are one-shot-and-exit; otherwise it shows the interactive menu (`run_menu`) when launched bare in a TTY (`sys.stdin.isatty()` and no action flag, unless `--no-menu`) or `--menu` is forced, else runs a single non-interactive sync. `build_importers()` constructs an importer per enabled/selected account via the `IMPORTERS` typeâ†’class registry. `run_concurrently()` runs a `work(importer)` callable across accounts in a `ThreadPoolExecutor` (up to 8) behind a live `ui.make_fetch_progress()` display (one spinner line per account, in config order, resolving to âœ“/âœ— + detail); `run_sync` uses it with `get_data()`, `run_doctor` with `diagnose()`, and both `run_reconcile`/`run_update_investments` with `get_balance()` (balances only â€” no transactions). `run_reconcile` then renders the table only; `run_update_investments` (investment accounts only, menu path) posts the Rendimento diffs and saves. The Rendimento/save/reconciliation logic is shared via `_maybe_post_rendimento` / `_save_and_render_summary` / `_render_reconciliation`. `run_sync` then processes results on the main thread in config order (queueing + optional `--update-investments` Rendimento adjustment), saves unless dry-run, and calls `ui.render_import_summary` + `ui.render_reconciliation`. Each account is wrapped in try/except (setup and fetch) so one failure doesn't abort the run. Flags: `--from`, `--dry-run`/`-n` (skip save), `-v`/`--show-transactions`/`--verbose` (per-account dump), `--debug`, `--reconcile` (balances-only reconciliation, exit), `--doctor`, `--list-accounts <ITEM_ID>`, `--update-investments`, `--menu`/`--no-menu`.
- `ui.py` - Presentation + interaction layer (rich + questionary). Owns the single shared `console` (imported by the importers for `--debug` logging during the concurrent fetch â€” rich's console is thread-safe and prints above an active `Progress`). Provides `make_fetch_progress()`, the menu/checklist/confirm prompts (`main_menu`, `select_accounts`, `ask_item_id`, `confirm_show_transactions`), and the `render_*` functions: `render_transactions` (opt-in dump, moved here from the importer), `render_import_summary` (panel), `render_reconciliation` (tables grouped by type), `render_doctor`, `render_discovery`. Also `money()`, `RECONCILE_TOLERANCE`, `RECONCILE_SECTIONS`. Deliberately imports **nothing** from `importers` (one-way dependency, so `base.py` can import `console` here without a cycle).
- `ynab_importer.py` - Orchestrates importing: `add_transactions()` filters already-fetched transactions by date and queues them (kept separate from the network fetch so accounts can be fetched concurrently); `get_transactions_from()` is the fetch-then-queue convenience wrapper; converts to YNAB format and batch saves.
- `importers/base.py` - `PluggyImporter` (auth, amountâ†’millicents, `get_data()` flow with abstract `_fetch_raw`/`_map_transaction` â€” runs in a worker thread so it does no console output beyond opt-in `--debug` lines emitted via the shared `ui.console` (`console.print`, thread-safe, prints above the live `Progress`); the caller renders the per-account dump afterwards via `ui.render_transactions()`. Also `_fetch_paginated` (loops all pages), `_fetch_balance` hook, `pluggy_balance` attr, and the doctor hooks: `diagnose()` (auth + `check_connection`, mirrors `get_data`, never raises) and `_fetch_item()` (`GET /items/{id}` for status/freshness); plus `get_balance()`/`_fetch_reconcile_balance()` for the reconcile-only flow (populate `pluggy_balance` via balance endpoints without pulling transactions â€” `AccountTransactionsImporter` overrides it to read `GET /accounts/{id}` directly since its `_fetch_balance` otherwise reuses the record from the transaction fetch). `AccountTransactionsImporter` fetches BANK/CREDIT accounts via paginated `GET /transactions?accountId` (balance via `GET /accounts/{id}`) and implements `check_connection` from the account record + its item.
- `importers/credit_card.py` - Credit card mapping. Nubank-specific cleanup (Apple, iFood, Uber, PayPal, etc.) is gated behind `bank == 'nubank'`; other banks use the raw description.
- `importers/checking_account.py` - Checking mapping. Cross-bank `document_payees` lookup (by payer/receiver CPF/CNPJ) runs for all banks; Nubank-specific text parsing is gated behind `bank == 'nubank'`.
- `importers/investment.py` - `PluggyInvestmentData`, **report-only**. `pluggy_source` is a Pluggy *item* id; `_fetch_raw` returns `[]` (no transactions to import), `_fetch_balance` lists `GET /investments?itemId`, keeps positions matching `investment_filter` (`_matches`: AND of optional `type`/`subtype`/`rate`, each value-or-list), and sums their `balance` into `pluggy_balance`. So investment accounts only feed the reconciliation table; the user fixes YNAB manually. `check_connection` (doctor) reports the matched position count + summed balance plus the item's status/freshness.
- `importers/data_importer.py` - Abstract base class for importers (`DataImporter` with `get_data()` method).
- `importers/transaction.py` - `Transaction` TypedDict shared by all importers.
- `importers/util.py` - YNAB budget/account lookup helper (`find_by_name`).

## Config files

- `.env` - Secrets only: `YNAB_TOKEN`, `YNAB_BUDGET`, `PLUGGY_CLIENT_ID`, `PLUGGY_CLIENT_SECRET`. Gitignored.
- `accounts.json` - List of accounts to sync. Each entry: `bank`, `type` (`checking`/`credit_card`/`investment`), `ynab_account` (YNAB account name), and `pluggy_account_id` (checking/credit_card) or `pluggy_item_id` (investment). Investment entries take an optional `investment_filter` (`type`/`subtype`/`rate`, value-or-list, AND-combined) selecting which of the item's positions count toward that account. Optional `enabled: false` to skip. Gitignored.
- `mappings.json` - Personal transaction mappings (Apple subscriptions, iFood restaurants, document-based payees). Gitignored. Optional - the script works without it.

## Key concepts

- Transactions use millicents (amount * 1000) for precision
- Pluggy auth uses one `PLUGGY_CLIENT_ID`/`SECRET` (one Pluggy application) for all banks/accounts; each bank is a connected *item* with its own accounts
- The `Transaction` TypedDict is the shared format between importers and the YNAB exporter
- A YNAB account is resolved by name (`ynab_account` in `accounts.json`); the importer writes to that account's id
- The `bank` field selects bank-specific parsing â€” only Nubank rules exist today; other banks fall back to generic behavior until per-bank rules are added
- Investments are NOT a Pluggy account type â€” they live under a separate `/investments` endpoint keyed by item id, and expose only a current balance per position (no transactions). They're report-only by default (summed/filtered, shown in the reconciliation table); `--update-investments` posts the diff vs YNAB as a single "Rendimento" transaction
- Mappings are loaded in sync.py and passed to importers via constructor
