# pluggy-ynab

Syncs Brazilian bank transactions to YNAB using Pluggy as the data provider.

## How to run

```bash
python sync.py                       # bare + interactive terminal -> opens the menu; otherwise (cron/piped) imports last 14 days
python sync.py --from 2026-01-01
python sync.py --dry-run             # fetch + summarize everything, don't save to YNAB
python sync.py -v                    # also print the per-account transaction dump (hidden by default; --show-transactions/--verbose)
python sync.py --dry-run --debug     # also print per-page fetch counts vs Pluggy's total
python sync.py --reconcile           # show only the balance reconciliation (Pluggy vs YNAB balances, no import), then exit
python sync.py --reconcile-commit    # reconcile & lock: mark cleared txns as reconciled for accounts whose CLEARED balance matches Pluggy, then exit
python sync.py --no-dedup-transfers  # import transfers between two synced accounts as two separate txns (dedup is ON by default)
python sync.py --doctor              # check each connection (auth + Pluggy reachability + freshness), then exit
python sync.py --list-accounts <ITEM_ID>   # discover Pluggy account ids for accounts.json, then exit
python sync.py --update-investments  # post the YNAB↔Pluggy diff per investment account as a "Rendimento" txn
python sync.py --no-menu             # force the non-interactive sync even in a terminal; --menu forces the menu anywhere
```

Requires a `.env` (secrets, see `.env.example`) and an `accounts.json` (list of accounts to sync, see `accounts.example.json`) at the project root. `mappings.json` is optional (see `mappings.example.json`).

The UI is built on **rich** (progress, panels, tables) + **questionary** (menus, with the highlighted option styled in cyan). Launched bare in a TTY, `sync.py` shows an **interactive menu**: Sync transactions / Update investments / Balance reconciliation / Reconcile & lock / Check connections / Quit. "Sync" runs an account-selection checklist + a post-sync "show transaction details?" prompt. "Update investments" is **investment-only** (fetches just the investment accounts and posts the Rendimento diff — the `--update-investments` *flag* instead layers the true-up onto a full sync of all accounts, for cron). Dry-run (`--dry-run`) and account discovery (`--list-accounts`) are flag-only (not in the menu). With no TTY (cron, piped) or any action flag, it runs non-interactively. The per-account transaction dump is **opt-in** (`-v` or the menu prompt) — `--dry-run` means "don't save" only and no longer implies `--debug`. During the concurrent fetch a live per-account line (spinner → ✓/✗ + count) is shown; the run ends with an Import Summary panel.

Every run ends with a balance-reconciliation table (grouped by type: checking / credit cards / investments) comparing each account's current Pluggy balance to its YNAB balance (cleared + uncleared), flagging match/MISMATCH (credit cards sign-inverted). Transaction fetching paginates through all pages (Pluggy caps a page at 500). Investment accounts are **report-only by default** (Pluggy exposes only position balances, no transactions); with `--update-investments` the per-account difference is posted as a "Rendimento" adjustment via `YNABTransactionImporter.add_adjustment`. The `import_id` is `REND-<date>-<time>-<acct8>` (**unique per run** — YNAB permanently remembers import_ids and rejects re-imports of a used one as a duplicate even after deletion, so a fixed daily id could never recreate a deleted Rendimento). Idempotency comes from the diff check instead: once an account is trued up, the next run sees `diff ≈ 0` and posts nothing (and after a save the cached YNAB balances are refreshed so a second update in the same menu session doesn't double-post).

**Transfer dedup (ON by default).** When the user moves money between two *synced* accounts, both feeds report it (debit on sender, credit on receiver), which would import as two unrelated transactions. After the concurrent fetch (where every account's mapped transactions are visible at once), `run_sync` calls `importers/transfer_match.find_transfer_pairs` to pair the two legs — same magnitude / opposite sign (exact milliunits, no tolerance, so it works on the post-mapping YNAB amount that already inverts credit-card sign), dated within `dedup_window_days` (default 3), across different accounts, and mutually unique (ambiguous ties are left as two normal txns; the `counterparty_document` CPF/CNPJ is a tiebreaker). Each pair is posted as a **single YNAB transfer** on the sending account via `YNABTransactionImporter.add_transfer` (`payee_id` = the receiving account's `transfer_payee_id`; YNAB auto-creates the linked mirror), and both raw legs are skipped from the normal import. `import_id` is the sender leg's raw Pluggy id, so overlapping windows re-detect the same pair and YNAB rejects the duplicate (idempotent). Both accounts must be enabled+selected to dedup; a transfer to an external (non-synced) account has only one leg and is imported normally. Disable with `--no-dedup-transfers`. Deduped transfers are listed in the Import Summary panel (shown on dry-run too).

**Reconcile & lock (`--reconcile-commit` / menu "Reconcile & lock").** Replicates the YNAB web reconcile *lock* via the API: for each selected account whose Pluggy balance matches YNAB's **cleared** balance within `RECONCILE_TOLERANCE` (credit cards sign-inverted), it marks every currently-`cleared`, non-deleted transaction as `reconciled` (the lock icon). Investment/tracking accounts are included on the same terms (no sign inversion — positions are positive asset balances); since their `Rendimento` adjustments (from `--update-investments`) also import **uncleared**, the user clears them in YNAB first so they become eligible to lock. It is **safe / lock-matched-only**: mismatched accounts are reported and skipped, never auto-adjusted (no balance-adjustment is ever posted — that decision was deliberate). Because imports land **uncleared**, only transactions the user has cleared in YNAB are eligible — a freshly-synced account usually shows a mismatch until cleared. The bulk update uses the YNAB `PATCH /budgets/{id}/transactions` endpoint directly (`YNABTransactionImporter.reconcile_transactions`, chunked) since `ynab_sdk==0.2.4` has no PATCH helper; the token/host are read from `self.ynab.client.config`. Idempotent: a re-run finds no `cleared` txns left and locks nothing. In a TTY it asks for confirmation first (`ui.confirm_reconcile_lock`); piped/cron runs skip the prompt (the lock-matched-only behaviour is safe).

## Project structure

- `sync.py` - Entry point / orchestration only (all presentation lives in `ui.py`). A `Session` holds creds + config and lazily connects the YNAB client (once per process, reused across menu iterations). `main()` parses args and dispatches: `--list-accounts`/`--doctor` are one-shot-and-exit; otherwise it shows the interactive menu (`run_menu`) when launched bare in a TTY (`sys.stdin.isatty()` and no action flag, unless `--no-menu`) or `--menu` is forced, else runs a single non-interactive sync. `build_importers()` constructs an importer per enabled/selected account via the `IMPORTERS` type→class registry. `run_concurrently()` runs a `work(importer)` callable across accounts in a `ThreadPoolExecutor` (up to 8) behind a live `ui.make_fetch_progress()` display (one spinner line per account, in config order, resolving to ✓/✗ + detail); `run_sync` uses it with `get_data()`, `run_doctor` with `diagnose()`, and both `run_reconcile`/`run_update_investments` with `get_balance()` (balances only — no transactions). `run_reconcile` then renders the table only; `run_update_investments` (investment accounts only, menu path) posts the Rendimento diffs and saves. `run_reconcile_commit` (the "Reconcile & lock" action) fetches balances, renders the read-only table, then per matched account (investments included) locks its cleared txns via `YNABTransactionImporter.fetch_cleared_transaction_ids` + `reconcile_transactions`, and renders `ui.render_reconcile_lock_summary`. The Rendimento/save/reconciliation logic is shared via `_maybe_post_rendimento` / `_save_and_render_summary` / `_render_reconciliation`. `run_sync` then runs `_dedup_transfers` (when enabled — pairs transfers and queues each via `add_transfer`, returning the skipped `consumed_ids`) before processing results on the main thread in config order (queueing the non-consumed txns + optional `--update-investments` Rendimento adjustment), saves unless dry-run, and calls `ui.render_import_summary` (with the deduped `transfers`) + `ui.render_reconciliation`. Each account is wrapped in try/except (setup and fetch) so one failure doesn't abort the run. Flags: `--from`, `--dry-run`/`-n` (skip save), `-v`/`--show-transactions`/`--verbose` (per-account dump), `--debug`, `--reconcile` (balances-only reconciliation, exit), `--reconcile-commit` (reconcile & lock, exit), `--no-dedup-transfers`, `--doctor`, `--list-accounts <ITEM_ID>`, `--update-investments`, `--menu`/`--no-menu`.
- `ui.py` - Presentation + interaction layer (rich + questionary). Owns the single shared `console` (imported by the importers for `--debug` logging during the concurrent fetch — rich's console is thread-safe and prints above an active `Progress`). Provides `make_fetch_progress()`, the menu/checklist/confirm prompts (`main_menu`, `select_accounts`, `ask_item_id`, `confirm_show_transactions`, `confirm_reconcile_lock`), and the `render_*` functions: `render_transactions` (opt-in dump, moved here from the importer), `render_import_summary` (panel, with an optional deduped-`transfers` section), `render_reconciliation` (tables grouped by type), `render_reconcile_lock_summary` (locked/skipped panel), `render_doctor`, `render_discovery`. Also `money()`, `RECONCILE_TOLERANCE`, `RECONCILE_SECTIONS`. Deliberately imports **nothing** from `importers` (one-way dependency, so `base.py` can import `console` here without a cycle).
- `ynab_importer.py` - Orchestrates importing: `add_transactions()` filters already-fetched transactions by date and queues them (kept separate from the network fetch so accounts can be fetched concurrently); `get_transactions_from()` is the fetch-then-queue convenience wrapper; converts to YNAB format and batch saves. `add_transfer()` queues one transfer leg (`payee_id` = receiving account's `transfer_payee_id`; YNAB mirrors it); `add_adjustment()` queues a synthetic balance adjustment (Rendimento); `eligible()` exposes the date-window filter for transfer dedup. Reconcile helpers: `fetch_cleared_transaction_ids()` (lists an account's currently-`cleared`, non-deleted txns) and `reconcile_transactions()` (bulk `cleared → reconciled` via a direct chunked `requests.patch` to the YNAB bulk-update endpoint, since `ynab_sdk==0.2.4` has no PATCH — token/host from `self.ynab.client.config`).
- `importers/base.py` - `PluggyImporter` (auth, amount→millicents, `get_data()` flow with abstract `_fetch_raw`/`_map_transaction` — runs in a worker thread so it does no console output beyond opt-in `--debug` lines emitted via the shared `ui.console` (`console.print`, thread-safe, prints above the live `Progress`); the caller renders the per-account dump afterwards via `ui.render_transactions()`. Also `_fetch_paginated` (loops all pages), `_fetch_balance` hook, `pluggy_balance` attr, and the doctor hooks: `diagnose()` (auth + `check_connection`, mirrors `get_data`, never raises) and `_fetch_item()` (`GET /items/{id}` for status/freshness); plus `get_balance()`/`_fetch_reconcile_balance()` for the reconcile-only flow (populate `pluggy_balance` via balance endpoints without pulling transactions — `AccountTransactionsImporter` overrides it to read `GET /accounts/{id}` directly since its `_fetch_balance` otherwise reuses the record from the transaction fetch). `AccountTransactionsImporter` fetches BANK/CREDIT accounts via paginated `GET /transactions?accountId` (balance via `GET /accounts/{id}`) and implements `check_connection` from the account record + its item.
- `importers/credit_card.py` - Credit card mapping. Nubank-specific cleanup (Apple, iFood, Uber, PayPal, etc.) is gated behind `bank == 'nubank'`; other banks use the raw description.
- `importers/checking_account.py` - Checking mapping. Cross-bank `document_payees` lookup (by payer/receiver CPF/CNPJ) runs for all banks; Nubank-specific text parsing is gated behind `bank == 'nubank'`. Also carries the counterparty CPF/CNPJ onto the mapped `Transaction` (`counterparty_document`) for transfer dedup.
- `importers/transfer_match.py` - `find_transfer_pairs(account_txns, window_days)` → `(pairs, consumed_ids)`. Pairs transfer legs across synced accounts (equal magnitude/opposite sign, within `window_days`, different accounts, mutually unique; `counterparty_document` breaks ties). `DEDUP_WINDOW_DAYS = 3`. Pure logic, no I/O.
- `importers/investment.py` - `PluggyInvestmentData`, **report-only**. `pluggy_source` is a Pluggy *item* id; `_fetch_raw` returns `[]` (no transactions to import), `_fetch_balance` lists `GET /investments?itemId`, keeps positions matching `investment_filter` (`_matches`: AND of optional `type`/`subtype`/`rate`, each value-or-list), and sums their `balance` into `pluggy_balance`. So investment accounts only feed the reconciliation table; the user fixes YNAB manually. `check_connection` (doctor) reports the matched position count + summed balance plus the item's status/freshness.
- `importers/data_importer.py` - Abstract base class for importers (`DataImporter` with `get_data()` method).
- `importers/transaction.py` - `Transaction` TypedDict shared by all importers (required fields + optional `counterparty_document` for transfer dedup, set only by the checking importer).
- `importers/util.py` - YNAB budget/account lookup helper (`find_by_name`).

## Config files

- `.env` - Secrets only: `YNAB_TOKEN`, `YNAB_BUDGET`, `PLUGGY_CLIENT_ID`, `PLUGGY_CLIENT_SECRET`. Gitignored.
- `accounts.json` - List of accounts to sync. Each entry: `bank`, `type` (`checking`/`credit_card`/`investment`), `ynab_account` (YNAB account name), and `pluggy_account_id` (checking/credit_card) or `pluggy_item_id` (investment). Investment entries take an optional `investment_filter` (`type`/`subtype`/`rate`, value-or-list, AND-combined) selecting which of the item's positions count toward that account. Optional `enabled: false` to skip. Optional top-level `dedup_window_days` (default 3) tunes how many days apart the two legs of a transfer may be. Gitignored.
- `mappings.json` - Personal transaction mappings (Apple subscriptions, iFood restaurants, document-based payees). Gitignored. Optional - the script works without it.

## Key concepts

- Transactions use millicents (amount * 1000) for precision
- Pluggy auth uses one `PLUGGY_CLIENT_ID`/`SECRET` (one Pluggy application) for all banks/accounts; each bank is a connected *item* with its own accounts
- The `Transaction` TypedDict is the shared format between importers and the YNAB exporter
- A YNAB account is resolved by name (`ynab_account` in `accounts.json`); the importer writes to that account's id
- The `bank` field selects bank-specific parsing — only Nubank rules exist today; other banks fall back to generic behavior until per-bank rules are added
- Investments are NOT a Pluggy account type — they live under a separate `/investments` endpoint keyed by item id, and expose only a current balance per position (no transactions). They're report-only by default (summed/filtered, shown in the reconciliation table); `--update-investments` posts the diff vs YNAB as a single "Rendimento" transaction
- Transfers between two synced accounts are deduped (ON by default) into a single YNAB transfer; matching is on the post-mapping milliunit `amount` (already credit-card sign-inverted), so the same rule covers checking↔checking and checking↔credit-card. Disable with `--no-dedup-transfers`
- Reconcile & lock (`--reconcile-commit`) marks an account's already-`cleared` transactions as `reconciled` only when its Pluggy balance matches YNAB's **cleared** balance within tolerance; it never posts a balance adjustment. Applies to all account types including investment/tracking accounts. Since imports (and `Rendimento` adjustments) stay `uncleared`, the user clears transactions in YNAB first, then locks
- A YNAB **transfer** is created by setting a transaction's `payee_id` to the *other* account's `transfer_payee_id` (on the `Account` model); YNAB auto-creates the linked leg. `cleared` status is one of `uncleared`/`cleared`/`reconciled`
- Mappings are loaded in sync.py and passed to importers via constructor
